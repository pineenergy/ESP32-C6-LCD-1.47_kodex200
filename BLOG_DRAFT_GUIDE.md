이 챕터에서는 결과를 먼저 보여주고, "예제에서 어디까지 확장했는가"를 짧게 설명하면 된다. 구현 디테일은 뒤 챕터에서 풀고, 여기서는 독자가 전체 목표를 바로 이해하도록 만드는 것이 중요하다.

- 이 프로젝트는 기존 데모 호출을 완전히 버리기보다, LVGL 초기화 이후 사용자 정의 화면을 올리는 방식으로 확장했다.

## 2. 개발 환경과 하드웨어 구성

개발은 ESP-IDF 5.4.1 기준으로 진행했고, 대상 보드는 ESP32-C6-LCD-1.47이다. 플래시는 UART 방식으로 진행했고, LCD와 RGB LED, SD, Wi-Fi가 한 프로젝트 안에서 함께 동작하도록 구성했다. 이 프로젝트의 장점은 별도의 센서 없이도 네트워크 데이터와 화면 출력을 바로 묶어볼 수 있다는 점이다.

이 부분은 기술 블로그답게 짧게 정리하는 것이 좋다. 환경 설정의 모든 과정을 길게 적기보다, 재현에 필요한 핵심 정보만 먼저 제시하고 상세 설치 과정은 별도 문서로 빼는 편이 읽기 좋다.

코드 설명 문장 예시:

- [.vscode/settings.json](.vscode/settings.json) 에는 현재 개발에 사용한 타깃 칩, ESP-IDF 경로, 포트, 플래시 방식이 정리되어 있다.
- 보드 설정은 간단해 보여도, 타깃 칩과 플래시 방식이 맞지 않으면 이후 빌드와 다운로드 과정에서 바로 문제가 발생한다.
- 따라서 이 챕터는 길게 설명하기보다, 실제 재현에 필요한 전제 조건을 빠르게 제시하는 역할로 두는 편이 좋다.

환경 요약:

- 🧰 ESP-IDF 5.4.1
- 📟 Target: esp32c6
- 🔌 Port: COM14
- 💾 Flash: UART

## 3. 원본 예제 구조 먼저 해부하기

프로젝트 구조를 이해하는 가장 빠른 방법은 [main/main.c](main/main.c) 를 먼저 보는 것이다. 이 파일에는 무선 기능 초기화, RGB 초기화, SD 초기화, LCD 초기화, LVGL 초기화, 그리고 최종 화면 생성까지의 흐름이 순서대로 들어 있다. 임베디드 프로젝트에서는 초기화 순서가 동작 안정성과 직접 연결되기 때문에, 기능을 추가할 때도 이 흐름을 크게 흔들지 않는 것이 중요했다.

이 챕터는 독자가 세부 구현에 들어가기 전에 전체 구조를 잡도록 도와주는 역할을 한다. 처음부터 네트워크 코드나 UI 코드만 보면 프로젝트가 분절돼 보이기 때문에, 앱 시작 흐름을 먼저 보여주는 편이 이해에 훨씬 유리하다.

- SD 초기화와 LCD 초기화 순서는 보드 특성상 의미가 있기 때문에, 이 부분을 바꾸는 경우에는 실제 동작 검증이 필요하다.
- 마지막 루프에서 lv_timer_handler 를 주기적으로 호출하는 구조는 LVGL 기반 UI가 계속 갱신되기 위한 핵심 조건이다.

## 4. KODEX200 데이터를 가져오기 위해 추가한 것들

실제 시세 표시기로 동작하려면 네트워크 계층이 먼저 안정적으로 붙어야 한다. 이 프로젝트에서는 [main/Wireless/Wireless.c](main/Wireless/Wireless.c) 에서 Wi-Fi 연결, SSID 후보 전환, 시간 동기화, HTTPS 요청과 같은 흐름을 담당한다. 단일 SSID만 사용하는 방식이 아니라 둘 이상의 후보를 두고 실패 시 전환하도록 만든 점은 실제 사용 환경을 고려한 부분이다.

또한 HTTPS 요청 전에 시간을 맞추는 처리도 들어 있다. 인증서 검증이 필요한 연결에서는 장치 시간이 맞지 않으면 요청이 실패할 수 있는데, 이런 지점은 데스크톱 환경보다 임베디드 환경에서 더 자주 드러난다. 그래서 이 챕터에서는 단순히 "값을 받아왔다"가 아니라, 그 전에 필요한 준비 작업도 함께 설명하는 편이 좋다.

코드 설명 문장 예시:

- [main/Wireless/Wireless.c](main/Wireless/Wireless.c) 는 Wi-Fi 연결 자체보다도, 실패했을 때 어떤 순서로 복구를 시도하는지까지 포함해 설계된 모듈이다.
- SSID 후보를 배열로 관리하는 방식은 설치 장소가 바뀌어도 펌웨어 수정 없이 대응할 여지를 만든다.
- 시간 동기화 로직은 HTTPS 통신의 전제 조건을 맞추기 위한 단계이며, 이 부분이 없으면 인증서 검증 실패로 이어질 수 있다.
- [main/Kconfig.projbuild](main/Kconfig.projbuild) 에서 종목 심볼과 갱신 주기를 설정값으로 분리해 둔 덕분에, 코드를 크게 손대지 않고도 동작 대상을 바꿀 수 있다.

소주제 후보:

- 📶 Wi-Fi 연결과 실패 대응
- 🔁 다중 SSID 전략
- 🕒 SNTP 시간 동기화
- 🔐 HTTPS 요청 전제 조건
- 📈 업데이트 주기 설정

## 5. 받은 데이터를 LCD에 어떻게 보여주도록 바꿨나

화면 구성은 [main/LVGL_UI/LVGL_Example.c](main/LVGL_UI/LVGL_Example.c) 에서 담당한다. 기존 LVGL 예제 화면을 그대로 쓰지 않고, KODEX200과 KODEX SMR의 가격과 변화량을 표시하는 정보형 UI로 바꾼 것이 핵심이다. 작은 LCD에서는 정보량보다 가독성이 중요하기 때문에, 숫자 포맷과 색상 구분, 갱신 상태 표시가 실제 체감 품질에 큰 영향을 준다.

이 챕터에서는 화면이 예뻐졌다는 얘기보다, 왜 이런 레이아웃과 포맷이 필요한지 설명하는 편이 좋다. 가격에 천 단위 구분기호를 붙이고, 등락값과 등락률을 분리해 표기하고, 로딩 상태를 별도로 두는 식의 선택은 모두 작은 화면에서의 읽기 쉬움을 높이기 위한 것이다.

코드 설명 문장 예시:

- [main/LVGL_UI/LVGL_Example.c](main/LVGL_UI/LVGL_Example.c) 는 예제성 위젯 화면을 정보 표시용 화면으로 재구성한 파일이다.
- 숫자 포맷팅 함수는 단순 출력이 아니라, 작은 디스플레이에서 값을 빠르게 읽히게 만드는 역할을 한다.
- 상승과 하락을 색으로 구분하는 로직은 값의 방향성을 한눈에 전달하기 위한 장치다.
- 갱신 주기와 진행 바를 함께 보여주는 방식은 현재 데이터가 최신인지 사용자가 스스로 판단할 수 있게 만든다.

## 6. 화면 말고도 상태를 어떻게 표현했나

이 프로젝트는 화면 출력 외에도 RGB LED를 보조 상태 표시로 활용한다. 숫자를 직접 읽지 않아도 현재 상태를 파악할 수 있도록, 데이터 유효 여부나 시세 방향을 LED 색에 반영하는 방식이다. 작은 임베디드 장치에서는 이런 보조 신호가 사용자 경험을 꽤 크게 개선한다.

블로그에서는 이 부분을 단순 장식 요소가 아니라, 상태 전달 채널을 하나 더 추가한 설계 포인트로 설명하는 편이 좋다. 특히 장치를 멀리서 볼 때도 상태를 알아차릴 수 있다는 점을 강조하면 기능적 의미가 더 분명해진다.

코드 설명 문장 예시:

- [main/RGB/RGB.c](main/RGB/RGB.c) 는 시세 변화값을 단순히 숫자로만 보여주지 않고 LED 색으로도 전달하도록 만든 모듈이다.
- [main/Wireless/Wireless.h](main/Wireless/Wireless.h) 의 공유 상태 값들은 UI와 LED가 같은 데이터 상태를 참조하게 만드는 연결점 역할을 한다.
- 이런 구성은 한쪽은 화면, 다른 한쪽은 보조 표시등처럼 동작하게 해 정보 전달 밀도를 높여준다.

사진 메모:

- 📸 LED 상태 사진 1장
- 📸 화면과 LED가 함께 보이는 사진 1장

## 7. 바이브 코딩으로 만들었지만, 어디까지는 반드시 검증했다

이번 프로젝트는 상당 부분을 바이브 코딩 방식으로 빠르게 밀어붙였다. 이미 동작하는 예제를 기반으로 필요한 기능을 덧붙이고, 결과를 보면서 구조를 정리하는 식으로 진행한 것이다. 이 방식은 초기 진입 속도가 빠르다는 장점이 있지만, 임베디드에서는 검증 없이 끝내기 어렵다.

특히 초기화 순서, 네트워크 실패 처리, 시간 동기화, 값이 없을 때의 UI 상태, 주기적 갱신 타이밍 같은 부분은 결국 사람이 직접 장치에서 확인해야 한다. 그래서 이 챕터는 "AI를 썼다"는 소개보다, "어디까지는 직접 검증했다"를 적는 쪽이 더 기술 블로그답다.

코드 설명 문장 예시:

- 이 프로젝트는 기존 예제를 빠르게 변형하는 데 AI의 도움을 받았지만, 실제 보드에서 초기화 순서와 통신 실패 케이스를 확인하는 과정은 따로 필요했다.
- 임베디드 환경에서는 코드가 컴파일되는 것과 장치에서 안정적으로 동작하는 것이 전혀 같은 의미가 아니다.
- 그래서 바이브 코딩은 출발 속도를 높이는 도구로는 유용하지만, 검증을 줄여주는 수단은 아니라는 점을 이번 작업에서 다시 확인했다.

짧은 회고 문단 예시:

> 이번 작업은 AI의 도움으로 구조를 빠르게 잡을 수 있었지만, 안정성은 결국 직접 확인해야 했다. 특히 Wi-Fi 연결 실패, HTTPS 요청 전 시간 동기화, LVGL 갱신 주기처럼 눈에 잘 드러나지 않는 부분은 장치 로그와 실제 화면을 함께 보며 검증할 필요가 있었다.

## 8. 만들고 나서 남은 개선 포인트

현재 상태만으로도 기본 목표는 달성했지만, 확장 여지는 충분하다. 종목 수를 늘려 여러 ETF를 순환 표시하도록 만들 수 있고, 버튼 입력을 붙여 사용자 조작이 가능하도록 확장할 수도 있다. 또한 저전력 모드나 재접속 전략, 화면 레이아웃 고도화 같은 부분도 다음 단계 주제로 적합하다.

블로그 마지막에는 이런 개선점을 3개 정도만 정리해도 글이 훨씬 완성형으로 보인다. 구현을 끝낸 뒤의 시야를 보여주기 때문이다.

코드 설명 문장 예시:

- 현재 구조는 특정 종목 값을 가져와 표시하는 데 초점이 맞춰져 있지만, 설정값 분리가 되어 있어 다른 종목으로의 확장은 비교적 단순하다.
- UI도 기본 정보 표시에는 충분하지만, 종목 수가 늘어나면 탭 구조나 카드형 레이아웃으로 발전시킬 여지가 있다.
- 네트워크 재시도와 저전력 정책은 실제 장치 운용 단계에서 가장 먼저 손볼 만한 개선 포인트다.

확장 아이디어:

- 📊 여러 종목 순환 표시
- 🕹️ 버튼으로 종목 전환
- 🔋 저전력 모드와 갱신 주기 조절
- ☁️ 데이터 소스 교체
- 📱 카드형 UI 개편

## 🧪 본문에 끼워 넣기 좋은 짧은 설명 문장 모음

- 이 프로젝트는 예제를 처음부터 다시 작성하기보다, 이미 동작하는 구조를 유지하면서 필요한 기능만 덧붙이는 방식으로 진행했다.
- 네트워크 연결과 UI 갱신은 각각 별개처럼 보이지만, 실제로는 데이터 유효 상태를 공유하면서 하나의 흐름으로 연결된다.
- 작은 LCD에서는 정보량보다 가독성이 더 중요하기 때문에, 숫자 포맷과 색상 구분이 생각보다 큰 역할을 한다.
- 임베디드 환경에서는 실패 케이스를 처리하는 코드가 정상 흐름만큼 중요하다.
- 바이브 코딩은 시작 속도를 높여주지만, 하드웨어가 얽히는 순간부터는 사람이 직접 확인해야 하는 구간이 반드시 생긴다.
